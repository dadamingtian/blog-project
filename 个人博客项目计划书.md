# 全栈个人博客项目计划书

## 1. 项目目标

构建一个可实际落地的全栈个人博客系统，包含前台博客展示与后台内容管理两部分。

- **前端技术栈**：Vue 3 + Tailwind CSS + Element Plus
- **后端技术栈**：Express
- **数据库**：MongoDB
- **目标要求**：
  - 支持博客文章发布、编辑、删除、浏览
  - 支持分类、标签、搜索、评论、归档
  - 支持后台登录与内容管理
  - 支持图片上传与文章封面
  - 项目完成后进行测试
  - 测试通过后创建本地 Git 仓库并提交一次初始化版本

本计划书的定位是：**让 AI 可以按阶段、按模块、按约束直接生成整个项目代码，而不是只生成零散片段。**

## 2. 项目定位与范围

### 2.1 用户角色

本项目默认包含两类角色：

1. **游客用户**
   - 浏览首页文章列表
   - 查看文章详情
   - 按分类/标签筛选文章
   - 搜索文章
   - 查看归档页
   - 发表评论
   - 查看关于页

2. **管理员**
   - 登录后台
   - 管理文章
   - 管理分类
   - 管理标签
   - 管理评论
   - 管理站点信息
   - 上传图片
   - 管理文章状态（草稿 / 已发布）

### 2.2 MVP 范围

优先完成以下最小可用版本：

- 前台博客展示
- 后台登录与文章管理
- 分类与标签管理
- 评论系统
- 搜索与归档
- 图片上传
- 基础 SEO 信息配置

### 2.3 二期增强功能（可选）

以下功能不作为第一阶段强制要求，但可以在主流程完成后补充：

- 点赞 / 阅读量统计
- Markdown 实时预览增强
- 草稿自动保存
- 文章置顶
- RSS 订阅
- 站内消息或评论审核通知
- 深色模式
- 访问统计看板
- 邮件通知
- Docker 部署

## 3. 整体架构设计

## 3.1 架构方案

采用前后端分离架构：

- `frontend/`：Vue 3 博客前端 + 后台管理前端
- `backend/`：Express REST API 服务
- `MongoDB`：存储文章、用户、评论、分类、标签、站点配置等数据
- 图片存储：第一阶段可采用后端本地静态目录存储，后续可升级为对象存储

### 3.3 开发原则

- 前后端职责清晰分离
- API 统一返回格式
- 所有重要业务逻辑都要有明确错误处理
- 后台接口必须进行登录校验与权限控制
- 表单校验前后端都要做
- 页面要兼顾桌面端体验
- 代码要具备可维护性，命名清晰，目录职责明确

## 6. API 设计

所有接口统一前缀建议：`/api`

统一响应格式建议：

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

错误格式：

```json
{
  "code": 1,
  "message": "error message",
  "data": null
}
```

## 6.1 认证接口

- `POST /api/auth/login` 管理员登录
- `GET /api/auth/profile` 获取当前管理员信息

## 6.2 文章接口

前台：
- `GET /api/articles` 获取已发布文章列表
- `GET /api/articles/:slug` 获取文章详情
- `GET /api/articles/search` 搜索文章
- `GET /api/articles/archive/list` 获取归档列表

后台：
- `POST /api/admin/articles` 新建文章
- `PUT /api/admin/articles/:id` 编辑文章
- `DELETE /api/admin/articles/:id` 删除文章
- `GET /api/admin/articles` 获取后台文章列表
- `GET /api/admin/articles/:id` 获取后台文章详情

## 6.3 分类接口

前台：
- `GET /api/categories`
- `GET /api/categories/:slug/articles`

后台：
- `POST /api/admin/categories`
- `PUT /api/admin/categories/:id`
- `DELETE /api/admin/categories/:id`
- `GET /api/admin/categories`

## 6.4 标签接口

前台：
- `GET /api/tags`
- `GET /api/tags/:slug/articles`

后台：
- `POST /api/admin/tags`
- `PUT /api/admin/tags/:id`
- `DELETE /api/admin/tags/:id`
- `GET /api/admin/tags`

## 6.5 评论接口

前台：
- `POST /api/comments` 新增评论
- `GET /api/comments/article/:articleId` 获取文章评论列表

后台：
- `GET /api/admin/comments`
- `PUT /api/admin/comments/:id/status`
- `DELETE /api/admin/comments/:id`

## 6.6 文件上传接口

- `POST /api/admin/upload`

## 6.7 站点设置接口

前台：
- `GET /api/site/config`

后台：
- `GET /api/admin/site/config`
- `PUT /api/admin/site/config`

## 6.8 仪表盘接口

- `GET /api/admin/dashboard`

## 8. 前端技术实现建议

### 8.1 推荐依赖

- `vue-router`：路由管理
- `pinia`：状态管理
- `axios`：接口请求
- `@element-plus/icons-vue`：图标
- `markdown-it` 或 `md-editor-v3`：Markdown 编辑 / 渲染
- `highlight.js`：代码高亮
- `dayjs`：日期处理

### 8.2 样式策略

- 使用 Tailwind CSS 负责基础布局、间距、响应式
- 使用 Element Plus 承担后台表单、表格、弹窗等组件
- 尽量保持前台风格简洁，后台风格偏管理系统

### 8.3 状态管理建议

建议至少拆分以下 Store：

- `authStore`：管理员登录信息、token
- `siteStore`：站点配置
- `articleStore`：文章列表与当前文章缓存（可选）

### 8.4 路由守卫

- 后台页面访问前检查 token
- 未登录访问后台则跳转 `/admin/login`
- 已登录访问登录页则跳转 `/admin/dashboard`

## 11. AI 生成代码时必须遵守的约束

1. **一次输出一个阶段的完整代码，不要只给思路。**
2. **每个阶段输出后，要保证代码能直接运行。**
3. **不要省略关键文件。**
4. **涉及环境变量时，必须同步输出 `.env.example`。**
5. **涉及数据库模型时，必须明确字段、类型、索引、默认值。**
6. **所有接口都要有明确的请求参数和返回结构。**
7. **前端页面必须包含基础加载态、空态、错误提示。**
8. **后台管理页面必须包含表单校验。**
9. **上传接口必须限制文件类型和大小。**
10. **所有管理端接口必须走 JWT 鉴权。**
11. **完成每一阶段后都要说明如何启动和验证。**
12. **最终阶段必须补充测试步骤与 Git 提交步骤。**

## 12. 测试计划

## 12.1 后端测试重点

- 登录接口是否正确返回 token
- 无 token 访问后台接口是否被拦截
- 文章新增 / 编辑 / 删除是否成功
- 分类与标签 CRUD 是否正常
- 评论新增与审核是否正常
- 上传接口是否正常返回文件地址
- 非法参数是否被正确拦截

## 12.2 前端测试重点

- 文章详情是否正确渲染 Markdown

## 15. 结论

这份计划书的核心作用不是单纯罗列功能，而是把整个博客项目拆成 **AI 可执行、可迭代、可验收** 的实施方案。

建议实际生成项目时，严格按照本计划书的阶段推进，不要跳步骤。这样可以显著减少 AI 一次性生成大项目时出现的：

- 文件遗漏
- 接口对不上
- 目录混乱
- 鉴权缺失
- 页面无法联调
- 最后无法启动

当你准备正式开始生成项目代码时，建议从“阶段 1：初始化项目骨架”开始逐步推进。
